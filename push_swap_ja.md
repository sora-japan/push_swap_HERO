# Push_swap

> *Swap_push だと自然な感じがしないから*

**概要**: このアクティビティでは、限られた命令セットを使いスタック上のデータをソートします。できるだけ少ない操作回数で目的を達成するために、様々なアルゴリズムを操作し、最適な解を選ぶ必要があります。

**このプロジェクトはちょうど2人のグループで取り組む課題です。**

## 第I章 序文

**コンピュータサイエンス**の神秘的な世界に、かつて Donald Knuth¹ という人物がいた。彼はアルゴリズムのスケール特性を語るために *Big-O 記法* を広めた。Big-O とは要するに、「*あなたのコードは永遠に速く動き続けるか、それともコーヒーを淹れて飲み、年老いるまで待ち続けるほど遅くなるか*」を上品に表現する方法だ。

伝説によれば、プログラミングの黎明期、プログラマーたちは Big-O を知らなかった。ただコードを走らせ、遅すぎたら *ハードウェア* のせいにしていた。そこに Big-O が登場し、あなたのアルゴリズムが数学的に証明可能なほど **最悪** であることを明らかにして、その言い訳を潰した。CPU ホイールにハムスターを何匹入れても無駄だということが証明されてしまった。

なぜこれが `push_swap` に関係するのか？ あなたは今、2つのスタックと極めて限られた操作しか持たず、寝不足な挿入ソートより速く数値をソートしようとしているからだ。Big-O は残酷な現実を突きつける：入力が大きくなるにつれて、華麗な O(n log n) 戦略は必ずお粗末な O(n²) を凌駕する……あなたが数学を無視して操作回数が爆発するのを眺めることを選ばない限り。

アルゴリズムを設計するとき、覚えておいてほしい：Big-O はあなたを怖がらせるためにあるのではなく、次のようなコードを書く人間になるのを防ぐためにある：

```
pb; pa; pb; pa; pb; pa; ...
```

これを1万回繰り返してチェッカーに嫌われ首をかしげるような人間に。**賢くソートせよ、遅くではなく。**

---
¹ そう、あの輝かしいひげと無限の忍耐を持つ男だ。

*Big-O はポップコーンのサイズ評価ではない…しかしそうだとしたら、O(n log n) が最高の選択だろう。*

## 第IV章 はじめに

**Push swap** プロジェクトは、非常にシンプルで明快なアルゴリズムプロジェクトだ：データをソートすること。

あなたには整数値のセット、2つのスタック、そして両スタックを操作するための操作セットがある。

目標は？ `push_swap` という C プログラムを書くこと。このプログラムは、引数として受け取った整数をソートする *Push swap 言語* の操作で構成された、**最小の操作列**を計算して標準出力に表示する。

簡単？

さあ、見てみよう……

## 第V章 目標

**このプロジェクトの目標**は、アルゴリズムの複雑性（計算量）を非常に具体的な形で体験することだ。

数値をソートすることは簡単だ。しかし2つのスタックと限られた操作だけで**素早く**ソートすることはまた別の話だ。完全にランダムなリストのソートと、ほぼソート済みのリストのソートもまったく異なる。

アルゴリズムの選択が、素早い勝利と終わりのない操作の連続との差を生むことをすぐに体感できるだろう。

### VI.2 ルール

- `a` と `b` という名前の2つのスタックを使う。
- 開始時：
  - スタック `a` には重複なしのランダムな量の負数・正数が含まれる。
  - スタック `b` は空。
- 目標はスタック `a` の数値を昇順にソートすること。そのために以下の操作が使える：

| 操作 | 説明 |
|------|------|
| **sa** (swap a) | スタック `a` の先頭2要素を入れ替える。要素が1つ以下なら何もしない。 |
| **sb** (swap b) | スタック `b` の先頭2要素を入れ替える。要素が1つ以下なら何もしない。 |
| **ss** | `sa` と `sb` を同時に行う。 |
| **pa** (push a) | スタック `b` の先頭要素をスタック `a` の先頭に移す。`b` が空なら何もしない。 |
| **pb** (push b) | スタック `a` の先頭要素をスタック `b` の先頭に移す。`a` が空なら何もしない。 |
| **ra** (rotate a) | スタック `a` の全要素を1つ上にシフトする。先頭要素が末尾になる。 |
| **rb** (rotate b) | スタック `b` の全要素を1つ上にシフトする。先頭要素が末尾になる。 |
| **rr** | `ra` と `rb` を同時に行う。 |
| **rra** (reverse rotate a) | スタック `a` の全要素を1つ下にシフトする。末尾要素が先頭になる。 |
| **rrb** (reverse rotate b) | スタック `b` の全要素を1つ下にシフトする。末尾要素が先頭になる。 |
| **rrr** | `rra` と `rrb` を同時に行う。 |

### VI.3 アルゴリズム要件

アルゴリズム的複雑性（*時間* と *空間*）への厳密な理解を強化するため、**4つの異なるソート戦略**を実装し `push_swap` プログラムに組み込まなければならない。プログラムは入力の状態に応じて実行時に戦略を選択できなければならない。

#### VI.3.1 複雑性モデルと操作制約

すべての戦略は C で実装され、ソートを行う Push_swap 操作列を生成しなければならない。つまり：

- C のアルゴリズムは入力を解析し、適切な操作列（`sa`, `sb`, `ss`, `pa`, `pb`, `ra`, `rb`, `rr`, `rra`, `rrb`, `rrr`）を生成する。
- 戦略の出力は、スタックをソートする操作列そのものだ。
- 計算量クラスを宣言する場合、それは古典的な配列ベースアルゴリズムの理論的計算量ではなく、**生成される Push_swap 操作数で測定されたコスト**を反映しなければならない。

#### VI.3.2 乱雑度メトリクス（必須）

この課題において、**disorder（乱雑度）** は 0 から 1 の値で、初期スタック `a` がソート済み状態からどれだけ離れているかを示す。

- 数値がすでに正しい順序なら disorder は 0。
- 最悪の順序なら disorder は 1。
- その間は部分的にソートされているが、まだ乱れている。

計算方法：スタック内のすべての数値ペアを見て、大きい数が小さい数の前に来るたびに *ミス* とカウントする。ミスが多いほど disorder は 1 に近づく。

```
function compute_disorder(stack a):
    mistakes = 0
    total_pairs = 0
    for i from 0 to size(a)-1:
        for j from i+1 to size(a)-1:
            total_pairs += 1
            if a[i] > a[j]:
                mistakes += 1
    return mistakes / total_pairs
```

> **disorder は操作を行う前に計測すること。**

#### VI.3.3 必要な戦略

1. **シンプルアルゴリズム（O(n²)）**
   O(n²) クラスのベースラインアルゴリズムを**少なくとも1つ**実装する。例：
   - 挿入ソートの適応
   - 選択ソートの適応
   - バブルソートの適応
   - シンプルな min/max 抽出法

2. **ミディアムアルゴリズム（O(n√n)）**
   O(n√n) クラスのアルゴリズムを**少なくとも1つ**実装する。例：
   - チャンクベースのソート（√n チャンクに分割）
   - ブロックベースの分割法
   - √n バケットを使ったバケットソートの適応
   - 範囲ベースのソート戦略

3. **コンプレックスアルゴリズム（O(n log n)）**
   O(n log n) クラスのアルゴリズムを**少なくとも1つ**実装する。例：
   - 基数ソートの適応（LSD または MSD）
   - 2スタックを使ったマージソートの適応
   - スタック分割を使ったクイックソートの適応
   - ヒープソートの適応
   - Binary Indexed Tree アプローチ

4. **カスタム適応型アルゴリズム（学習者の設計）**
   計測された `disorder` に応じて異なる内部手法を選択する**適応型**戦略を設計する。特定の名前付きアルゴリズムに縛られる必要はなく、内部技術は完全に自由だ。ただし設計は以下の**計算量ターゲット**（Push_swap 操作モデルで）を満たすこと：

   | 乱雑度の範囲 | 要求計算量 |
   |---|---|
   | disorder < 0.2（低乱雑） | O(n²) |
   | 0.2 ≤ disorder < 0.5（中乱雑） | O(n√n) |
   | disorder ≥ 0.5（高乱雑） | O(n log n) |

   リポジトリ（例：`README.md`）に以下を文書化すること：
   - 閾値の根拠
   - 各レジームで使用する内部技術
   - Push_swap モデルにおける時間・空間の計算量の簡単な上界の論拠

### VI.5 push_swap プログラム

| 項目 | 内容 |
|------|------|
| プログラム名 | `push_swap` |
| 提出ファイル | `Makefile`, `*.h`, `*.c` |
| Makefile | `NAME`, `all`, `clean`, `fclean`, `re` |
| 引数 | スタック a：整数のリスト |
| 使用可能な外部関数 | `read`, `write`, `malloc`, `free`, `exit`、自分が書いた `ft_printf` または同等のもの |
| libft の使用 | 可 |
| 説明 | スタックをソートする |

プロジェクトは以下のルールに従うこと：

- リンクのないコンパイルを行う `Makefile` を提出すること。
- **グローバル変数は禁止。**
- 以下の引数を取る `push_swap` という名前のプログラムを書くこと：
  - スタック `a`：整数のリスト（最初の引数がスタックの先頭）
  - オプションの**戦略セレクタ**：
    - `--simple`：O(n²) アルゴリズムを強制使用
    - `--medium`：O(n√n) アルゴリズムを強制使用
    - `--complex`：O(n log n) アルゴリズムを強制使用
    - `--adaptive`：disorder に基づく適応型アルゴリズムを強制使用（セレクタ未指定時のデフォルト動作）
- スタック `a` をソートするための最小の Push_swap 操作リストを表示すること（最小値が先頭）。
- 操作は `\n` で区切り、それ以外は何も出力しないこと。
- 各アルゴリズムに主張する計算量クラスはこのモデルで有効でなければならない。
- 戦略選択は**すべての有効な入力**に対して機能すること。どのセレクタフラグも入力サイズや乱雑度に関わらず動作すること。
- パラメータが指定されない場合、プログラムは何も表示せずプロンプトに戻ること。
- エラー時は標準エラーに `"Error"` と `\n` を表示すること。エラーの例：整数でない引数、有効範囲外の整数、重複値。
- バイナリには4つの戦略すべて（Simple O(n²)、Medium O(n√n)、Complex O(n log n)、Adaptive）が組み込まれていること。選択された戦略名と計算量クラスは `--bench` モードで利用可能であること。
- オプションの**ベンチマークモード**（`--bench`）は、ソート後に以下を `stderr` へ出力すること：
  - 計算された `disorder`（小数点以下2桁の %）
  - 使用した戦略名とその理論的計算量クラス
  - 操作の総数
  - 各操作種別のカウント（`sa`, `sb`, `ss`, `pa`, `pb`, `ra`, `rb`, `rr`, `rra`, `rrb`, `rrr`）

#### VI.5.1 使用例

> 注：**[bench]** が先頭の行はオプションのベンチマークモードが `stderr` に出力するメッセージ。操作ストリームは `stdout` に残る。

**デフォルト実行：**
```
$> ./push_swap 2 1 3 6 5 8
ra
pb
rra
pb
...
pa
pa
```

**デフォルト選択（--adaptive）と操作数カウント：**
```
$> ARG="4 67 3 87 23"; ./push_swap --adaptive $ARG | wc -l
```

## 第VII章 README 要件

Git リポジトリのルートに `README.md` ファイルを用意すること。目的は、プロジェクトに不慣れな人（ピア、スタッフ、採用担当者など）がプロジェクトの内容、実行方法、関連情報を素早く理解できるようにすること。

## 第VIII章 ボーナスパート

このプロジェクトはシンプルなため、追加機能の余地は限られている。しかし、自分自身のチェッカーを作ってみては？

> `checker` プログラムにより、`push_swap` が生成した操作列が実際にスタックを正しくソートするかを確認できる。

> **注意**：ボーナスパートは必須パートが**完璧**な場合のみ評価される。完璧とは、必須パートが完全に完成し、エラーなく機能していること。このプロジェクトでは、すべてのベンチマークを例外なく通過することを意味する。必須要件をすべて満たしていない場合、ボーナスパートはまったく評価されない。

### VIII.1 checker プログラム

| 項目 | 内容 |
|------|------|
| プログラム名 | `checker` |
| 提出ファイル | `*.h`, `*.c` |
| Makefile | `bonus` |
| 引数 | スタック a：整数のリスト |
| 使用可能な外部関数 | `read`, `write`, `malloc`, `free`, `exit`、自分が書いた `ft_printf` または同等のもの |
| libft の使用 | 可 |
| 説明 | ソート操作を実行する |

- スタック `a` を整数のリストとして引数に取る `checker` という名前のプログラムを書くこと（最初の引数がスタックの先頭。順序に注意）。引数がない場合は停止して何も表示しない。
- 標準入力から操作を読み込む。各命令は `\n` で終わる。すべての命令を読み込んだ後、受け取ったスタックに対してそれらを実行する。
- 命令実行後にスタック `a` がソートされ、スタック `b` が空なら、標準出力に `"OK"` と `\n` を表示する。
- それ以外の場合は標準出力に `"KO"` と `\n` を表示する。
- エラー時は**標準エラー**に `"Error"` と `\n` を表示する。エラーの例：整数でない引数、整数の範囲外の引数、重複、存在しない命令・不正なフォーマットの命令。

**使用例：**
```
$> ./checker 3 2 1 0
rra
pb
sa
rra
pa
OK
$> ./checker 3 2 1 0
sa
rra
pb
KO
$> ./checker 3 2 one 0
Error
$> ./checker "" 1
Error
$>
```

> **提供されたバイナリと全く同じ挙動を再現する必要はない。** エラー処理は必須だが、引数のパース方法は自由に決めてよい。

## 第IX章 提出とピア評価

通常通り Git リポジトリに課題を提出すること。評価（ディフェンス）ではリポジトリ内の作業だけが評価される。ファイル名が正しいか必ず確認すること。

### グループプロジェクトの提出要件

- 両学習者がリポジトリのコントリビューターとしてリストされていること。
- `README.md` に両学習者の貢献が明確に記載されていること。
- ピアレビューのディフェンスには両学習者が出席すること。
- 各学習者がコードのどの部分も説明・防衛できること。

評価中に、プロジェクトの**簡単な修正**が求められることがある。これには、挙動のマイナーな変更、数行のコードの追加・書き直し、簡単な機能の追加などが含まれる。

この手順はすべてのプロジェクトに適用されるわけではないが、評価ガイドラインに記載されている場合は準備しておくこと。

この手順は、プロジェクトの特定部分を実際に理解しているかを確認するためのものだ。修正は任意の開発環境で行うことができ、特定の時間枠が定義されていない限り数分以内に完了できるものだ。

詳細（範囲、対象など）は**評価ガイドライン**に明記され、同じプロジェクトでも評価ごとに異なる場合がある。
